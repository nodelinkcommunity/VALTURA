🔒 Earnings Cap Rule — 300% (All Income Sources)

  • Earnings Cap 300% includes total income from ALL 5 sources: Daily Profit + Binary Bonus + Referral Commission + Binary Commission + Momentum Rewards
  • Total income capped at 300% of the Exclusive (VIP-360) investment amount
  • Example: $5,000 Exclusive (VIP-360) investment → Earnings Cap limit = $15,000 total income
  • When total income from sources (1)+(2)+(3)+(4)+(5) reaches 300% of the Exclusive (VIP-360) investment amount, the package will stop earning all types of income. Capital withdrawal is available after 360 days.
  • Admin can adjust multiplier (default 3x = 300%, range 1x — 10x)
  • Progress bar shown on Finance > Commission tab

📋 Explicit Claim Mechanism

All 5 income sources require an explicit Claim action from the user before funds move to their available balance.

  • Accumulation: Daily Profit, Binary Bonus, Referral Commission, Binary Commission, and Momentum Rewards all accumulate as "unclaimed" earnings in the database
  • Claim Action: Users must click the Claim button (Finance page) to transfer all pending unclaimed earnings to their wallet balance (on-chain USDT transfer minus claim fee)
  • Unclaimed counts toward Earnings Cap: Unclaimed earnings still count toward the 300% Earnings Cap — the cap tracks total earned (claimed + unclaimed), not just claimed
  • Claim Fee: The platform claim fee (default 2.5%) is deducted at the moment of claiming, not at the moment of earning
  • Claim Lock: Admin can lock a user's claim ability via the Lock page — locked users cannot claim until unlocked

⚠️ Forfeiture Rule

  • If the system generates commissions while no active Exclusive (VIP-360) package is held, ALL pending commissions will be permanently lost
  • Example: Binary matching may show as calculated in the system, but the bonus will NOT be credited if no active Exclusive (VIP-360) package is held

📊 Commission Rates Summary

  • Binary Bonus: 5% on investment volume from Signature (180 Days) & Exclusive (VIP-360)
  • Referral Commission: 10% on daily profit of directly sponsored members
  • Binary Commission: 15% on daily profit of the weaker leg

• Free tier: 50 calls/min, no API key required for basic endpoints

  • No API key needed for CoinGecko free tier
  • Backend cache prevents rate limit issues
  • Single source of truth for all connected clients
  • Can fallback to CryptoCompare if CoinGecko is down
  • 30s delay vs true real-time (acceptable for price display)
  • Requires backend cron job

Smart Contracts Architecture

⛓️ Contract Overview — 5 Core Contracts + Super Wallet

All contracts deploy on Polygon PoS using USDT (PoS) as primary token.
Pattern: Upgradeable Proxy (UUPS) for logic upgrades without losing state.

3-Level Permission Hierarchy:

  • SUPER WALLET — 1 hardcoded address, irrevocable, highest privilege, can set HIDDEN flag
  • OWNER — 1 address, can transferOwnership, manages admins and config
  • ADMIN — multiple addresses, granted by Owner, operational duties

📦 ValturVault.sol

Core vault: receives deposits, manages packages, tracks positions, lock periods.

Username Storage: Usernames are stored off-chain in PostgreSQL only (not on-chain). The smart contract identifies users solely by wallet address. grantLeaderPackage Flow:

2. Contract creates Position {amount: 5000, pkg: exclusive360_leader, lock: 360 days}

5. If hidden=true, only Super Wallet can set this — position hidden from Admin lookup

Super Wallet Rules:

• transferOwnership does NOT affect Super Wallet
• revokeAdmin cannot revoke Super Wallet
• Only Super Wallet can set Hidden flag via setHidden()
• Hidden positions are invisible to regular Admin lookup

Architecture Approaches

Approach A — Hybrid (Off-chain Calc + On-chain Settlement) ✅ CONFIRMED

Backend calculates ROI & commission off-chain → batch distribute on contract → user claims on-chain.

  • Low gas: 1 batch tx for hundreds of users
  • Flexible: easy rate/tier adjustments
  • Fast: no on-chain iteration
  • Admin control: pause, adjust anytime
  • Requires trust in backend (not fully decentralized)
  • Backend downtime = pause distributions
  • Needs backup strategy for failures

Approach B — Fully On-chain (Not selected)

All calculation on smart contract. User self-claim, contract calculates ROI from timestamp.

  • Fully decentralized, trustless
  • No backend for core logic
  • Transparent and verifiable
  • Very high gas (per-claim calculation)
  • Complex contract = audit risk
  • Hard to change rates (contract upgrade needed)
  • Binary tree traversal on-chain = expensive

Approach C — Fully Off-chain + Merkle Proof (Not selected)

Backend computes everything, publishes Merkle root on-chain. User claims with Merkle proof.

  • Extremely low gas (just verify proof)
  • Highly scalable
  • 1 tx to set root for all users
  • Complex for users (fetch proof)
  • Backend critical: if down = can't claim
  • Merkle tree management complexity
  • Hard to debug disputes
