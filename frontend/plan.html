2. Deposit Flow (User Buys Package)

1
Frontend: User selects package + amount → Calls USDT.approve(Vault, amount) on-chain
2
Frontend: Calls Vault.deposit(amount, lockDays, tier, pkgType) on-chain → USDT transfers to Vault, position created on-chain
3
Frontend: Sends txHash (of vault.deposit, NOT approve) → Backend API POST /api/invest/deposit
4
Backend: Verify on-chain:
— txHash exists and confirmed
— Parse event Deposited(address, uint256, uint8, uint8)
— Verify: user wallet matches, amount matches, to = Vault address
— Get on_chain_index = getUserPositionCount(wallet) - 1
5
Backend: Save to DB: position with on_chain_index, tx_hash, status=active
— Update binary tree volumes
— Calculate Binary Bonus X% (from db.getConfigValue('comm_binary_bonus'), default 5%) if Signature/Exclusive
6
Backend: Auto-split deposit to funds:
— Read fund_trading_pct from config (default 85%)
— Call Vault.withdrawToTradingFund(amount × tradingPct / 100)
— Remaining stays in Vault = Reward & Redeem Fund
Backend rejects fake txHash or approve txHash (no Deposited event).
🇷🇴 Română Utilizatorul selectează pachet + sumă → Approve USDT → Apel Vault.deposit() on-chain → Frontend trimite txHash la Backend → Backend verifică on-chain (eveniment Deposited, wallet, sumă) → Salvează în DB → Împarte automat: 85% la TradingFunds, 15% rămâne în Vault (Reward Fund). Procentele se citesc din config Admin (nu sunt fixe).

3. Daily ROI Flow (Cron Job)

1
Backend Cron: Calculate daily ROI for all active positions (amount × dailyRate%)
2
Backend: Call on-chain ROIDistributor.distributeROI(users[], amounts[], epoch)
3
Backend: Calculate commissions: Referral X% (from comm_referral), Binary Y% (from comm_binary), Momentum
4
Backend: Call on-chain CommissionPayout.distributeCommissions(users[], types[], amounts[], epoch)
5
Backend: Save to DB: earnings, commission records. On-chain and DB 100% synced.
🇷🇴 Română Cron zilnic: Calculează ROI pentru toate pozițiile active → Distribuie on-chain → Calculează comisioane (Referral X%, Binary Y%, Momentum) → Distribuie on-chain → Salvează în DB. On-chain și DB sincronizate 100%.

5. Redeem Flow (User Withdraws Capital)

Core principle: User clicks Redeem → Backend only (pending). On-chain ONLY happens when Admin Approves. Package stays active, keeps earning ROI until approved.

Step 1: User Requests Redeem

1
Frontend: User clicks "Redeem Funds" on expired lock package
— NO on-chain call
— Only sends POST /api/invest/redeem { positionId } to backend
— Package stays 100% active on-chain, keeps receiving Daily ROI
2
Backend:
— Creates redemption order: status = 'pending'
— Position status UNCHANGED, stays 'active'
— Appears in Admin > Redeem Orders for review

Step 2a: Admin APPROVES

3
Admin: Clicks "Approve" in Redeem Orders
4
Backend automatically executes ON-CHAIN (3 steps):
a) Vault.requestRedemption(user, posId) → deactivates position, reduces TVL
b) RedemptionManager.createOrder(user, posId, amount)
c) RedemptionManager.approveRedemption(orderId) → USDT sent to user wallet (minus fee, from Vault via transferFrom)
— Redeem fee = parseFloat(db.getConfigValue('fee_redeem')) % (NOT hardcoded)
5
Backend updates DB:
— position.status = 'completed'
— redemption.status = 'approved'
— redemption.tx_hash = TX hash of refund transaction
6
User Frontend — Redeemed Packages:
— Status: ✅ Done
— Button: "See Transaction" → Polygonscan link to refund TX
— Shows: Package name, Amount, Fee %, Net Amount, Request date, Process date

Step 2b: Admin REJECTS

3
Admin: Clicks "Reject" in Redeem Orders
4
Backend:
— NO on-chain changes — package stays active, keeps earning ROI
— redemption.status = 'rejected'
— position.status stays 'active' (unchanged)
— User can submit Redeem request again anytime
Important notes:
— No reactivatePosition() needed — on-chain is unchanged until admin approves
— Exclusive Leader Access (admin grant): CANNOT Redeem (frontend hides button, backend rejects)
— Packages with unexpired lock: CANNOT Redeem (frontend disables button)
— Redeem fee from db.getConfigValue('fee_redeem') — NOT hardcoded
🇷🇴 Română Utilizatorul cere Redeem → doar backend (pending). On-chain doar la Aprobare Admin. Aprobare: Backend dezactivează pe chain + trimite USDT la utilizator (minus comision). Respingere: Nimic on-chain, pachet rămâne activ. Nu e nevoie de reactivatePosition(). Leader packages nu pot fi răscumpărate.

🚨 MaxOut (Earnings Cap) — Detailed Sync

MaxOut = Maximum income per package = maxout_% × Investment amount. Applies to ALL packages, not just VIP.

💡 MaxOut Logic:
— Non-VIP packages: MaxOut calculated on Daily ROI only
— VIP packages: MaxOut calculated on ALL income (ROI + Bonus + Commission + Momentum)
— When MaxOut reached → position stops receiving ROI
— Lost income recorded in earnings_lost table
